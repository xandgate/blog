## Overview

A large regulated enterprise platform was running a custom Drupal data visualization module that had accumulated significant technical debt. The module handled charts and interactive data displays across the site — functional, in production, and quietly becoming harder to maintain with every passing month.

The refactor addressed four concrete problems: an unsustainable fork of a community module, an outdated charting library, inaccessible chart output, and an editorial interface that required developer involvement for routine content tasks. A fifth issue — complex charts on small screens — was deliberately resolved as a governance decision rather than a code problem.

## The Problem

The module had been built by forking the Drupal Charts contributed module. For anyone outside the Drupal ecosystem: the contrib module community is one of Drupal's core strengths. When you fork a community module, you diverge from it. Security patches, improvements, and compatibility updates from the community no longer reach you automatically — you're responsible for manually porting anything you need into a codebase that has drifted progressively further from the original.

By the time of this engagement, the fork was multiple major versions behind. The underlying Chart.js library was outdated. Charts failed accessibility standards with no alternative for screen reader or keyboard-only users. And the editorial interface had been designed around the data model rather than around how content editors think.

None of this was visible to end users. The charts rendered. The website worked. The debt was invisible, accumulating.

## Solution

![Chart and “View as table” toggle: same data, accessible alternative](/images/projects/data-viz/screenshotsChartsMobile.png)

### Stop Hacking Contrib

The architectural fix: retire the fork entirely and treat the Charts contributed module as a proper dependency rather than a base to modify.

The new module extends Charts without touching it. Security patches now apply cleanly. Community improvements flow downstream automatically. Version upgrades follow the standard Drupal update process rather than requiring a bespoke merge. The platform moved from "someone needs to watch this carefully" to "it handles itself."

This change had zero visible impact on end users. It has the highest long-term value of anything delivered in the project.

### Modernize to Current Chart.js

Moving to the current major version of Chart.js wasn't a cosmetic upgrade. It brought improved rendering performance, better responsive behavior, and access to chart types and configuration options unavailable in the previous version. It also removed a dependency on a release that upstream had effectively superseded.

### Accessibility: View Chart as Table

The previous implementation had no accessible alternative to the visual chart. For users relying on screen readers — or users who simply prefer structured data — the information was inaccessible in any meaningful sense.

The refactor introduced a **View Chart as Table** toggle: a single button that switches the visualization to a clean data table presenting the same underlying information. Same data, different format, no information loss.

For Section 508 and WCAG compliance purposes, this is the difference between passing an audit and failing one. For users who rely on assistive technology, it is the difference between accessing the content and not.

### Editorial Experience

![Charts module brings UX enhancements](/images/projects/data-viz/table-for-filling-out-chart-data.png)

The previous configuration interface was built for engineers. Content editors managing charts had to navigate options designed around the underlying data model, not around the editorial task.

The rebuilt interface is organized around how editors think about charts: what they are trying to communicate, what data goes where, what the chart should show. The result is faster content updates, fewer configuration errors, and significantly reduced dependency on developer involvement for routine editorial work.

## The Governance Decision

Complex charts on small screens are a genuine UX problem. When a visualization has many data points, long axis labels, or fine-grained detail, a 375px viewport and a dense bar chart are fundamentally in tension. No amount of responsive CSS fully resolves that.

The temptation in a refactor like this is to try to solve everything in code. The right call was not to.

This is a design and content governance question, not a module question. The appropriate solution is an editorial standard: guidance on chart complexity thresholds, recommended data density limits per chart type, and a review process for visualizations intended for mobile audiences. Attempting to handle it in the module produces brittle workarounds. Handling it in governance produces a repeatable standard that improves every future chart, not just the cases the module happens to intercept.

Identifying the boundary between what belongs in code and what belongs in a style guide is part of the work.

## Technologies

- **Drupal 10** — enterprise CMS platform
- **Charts (contrib module)** — community-maintained charting integration, used as a dependency rather than a fork
- **Chart.js** — underlying JavaScript charting library, upgraded to current major version
- **PHP / Drupal module API** — custom extension layer
- **ARIA / WCAG 2.1** — accessibility standards implementation

## Impact

- Eliminated the custom module fork and its associated maintenance burden
- Restored community upgrade path for the Charts module and Chart.js
- Achieved WCAG 2.1 AA compliance for all data visualizations
- Reduced editorial dependency on developer involvement for chart content updates
- Established governance framework for chart complexity on small screens

## Outcome

The module is in production. The charts render. The website works.

The difference is that the platform is no longer one Chart.js security release away from a manual merge exercise, one accessibility audit away from a remediation backlog, or one content editor away from a support ticket.

The most durable engineering work is often the kind that removes future problems before anyone knows they were coming.
