## Overview

A regulated enterprise platform was managing SAML Identity Provider configuration the same way it managed most other Drupal configuration: in the database, synced via config management. That works for most settings. For SSO endpoints, it was a deployment hazard.

The solution uses `ConfigFactoryOverrideInterface` to inject Key-module-resolved values into Drupal's config system at runtime — without modifying the upstream `samlauth` module. A companion upstream contribution extends the module's existing `key:` prefix pattern to the IdP string fields that were missing it.

## The Problem

Drupal's config management assumes settings are environment-agnostic. For most of a site's configuration, that's true. But SAML Identity Provider settings are fundamentally environment-specific: `idp_entity_id`, the SSO endpoint URL, the SLO endpoint URL. Development, staging, and production each point to different IdP instances — or should.

When a team copies a production database to staging — a routine operation — they carry production IdP configuration into a non-production environment. In the best case, authentication fails loudly. In the worst case, it succeeds silently, and staging sessions authenticate against the live enterprise IdP.

Teams typically reach for one of three workarounds, each with its own failure mode.

The `settings.php` override is the quickest:

```php
$config['samlauth.authentication']['idp_entity_id'] = getenv('SAML_IDP_ENTITY_ID');
```

It works, but it bypasses Drupal's config layer entirely. The config inspector shows the database value, not the live value. The override is invisible to anyone reading the configuration admin UI. `settings.php` becomes a graveyard of one-off workarounds that nobody audits and everyone inherits.

The `config_split` module is the more structured answer — it lets you define environment-specific config splits that activate based on a flag, keeping environment config out of your main sync directory. It is genuinely useful for feature flags and environment-scoped settings, but it fast becomes cumbersome for secrets management in practice. You are still storing the actual IdP values in configuration files, which means they live in or adjacent to version control. Rotating a credential means a config export, a commit, and a deploy. For values that need to stay out of the repository entirely — IdP certificates, endpoint URLs that vary per client — `config_split` solves the wrong problem.

The right tool for secrets is a secrets manager. In Drupal's ecosystem, that's the Key module.

## The Architecture

### ConfigFactoryOverrideInterface

The production solution is a lightweight custom module implementing Drupal's `ConfigFactoryOverrideInterface`. When Drupal loads `samlauth.authentication`, this module intercepts the config object and injects Key-resolved values before `samlauth` ever reads them.

```php
public function loadOverrides($names) {
  if (!in_array('samlauth.authentication', $names)) {
    return [];
  }
  return [
    'samlauth.authentication' => [
      'idp_entity_id'              => $this->resolveKey('saml_idp_entity_id'),
      'idp_single_sign_on_service' => $this->resolveKey('saml_idp_sso_url'),
      'idp_single_log_out_service' => $this->resolveKey('saml_idp_slo_url'),
    ],
  ];
}
```

The module makes zero changes to upstream `samlauth`. When samlauth ships a new release, the override module survives the update unchanged. The separation is explicit: samlauth handles SAML protocol logic; this module handles environment-specific configuration injection. That boundary matters for a platform under active maintenance.

### The Key Module

Rather than calling `getenv()` directly, the override resolves values through Drupal's Key module. Key entities are named secrets with swappable providers. An entity using the environment provider reads from `$_ENV` at runtime. The same entity can be switched to read from HashiCorp Vault, a file on disk, or a different variable — without changing PHP.

This matters in regulated environments. Credential rotation changes a Key entity's provider configuration, not code. The secret source is auditable through Drupal's admin UI. And local development, CI, staging, and production all use the same Key entity names with different providers per environment — no special-casing in application code.

### The Gap in Upstream samlauth

The `samlauth` module has supported the Key module for certificate fields since version 3.3: store `key:my-cert-name` as the config value and `SamlService::reformatConfig()` resolves it at runtime. The gap was that IdP string fields — entity ID, SSO URL, SLO URL — didn't support the same `key:` prefix pattern. Only cert values did.

The `ConfigFactoryOverrideInterface` module bridges this for production. But it means every team solving this problem has to write the same custom module. That is the upstream gap worth addressing.

## Local Testing

SAML development without a local identity provider is a productivity drain — every auth test requires a live IdP. The development environment runs SimpleSAMLphp as a mock IdP via a DDEV custom docker-compose service. Developers can exercise the full authentication loop locally without touching production credentials. Mock IdP, Key entities reading from DDEV environment configuration, local Drupal: the same configuration structure that deploys to staging and production, running entirely on a laptop.

## The Upstream Contribution

The custom module is the correct production approach when stability guarantees matter. It is not the right long-term answer for the ecosystem.

The upstream contribution — [issue #3578383](https://www.drupal.org/project/samlauth/issues/3578383) against `samlauth` — extends the existing `key:` prefix pattern in `SamlService.php` to the three IdP string fields and adds corresponding form UI. The implementation adds a `resolveKeyValue()` static helper that mirrors the existing cert resolution logic, three callsites in `reformatConfig()`, and conditional form widgets controlled by `#states`. If it lands, teams configure environment-aware SAML directly through the samlauth admin interface — no custom module required.

The measure of whether the upstream contribution succeeds is simple: the custom override module becomes unnecessary.

## Outcome

Database copies no longer carry wrong IdP configuration into non-production environments. The authentication layer resolves from the environment at runtime — a database refresh does not require a manual reconfiguration step.

The pattern transfers. Any Drupal configuration that varies by environment and involves credentials — OAuth client secrets, API keys, encryption keys — can use the same Key module and `ConfigFactoryOverrideInterface` approach. Getting SAML right was worth doing for its own sake. The architectural pattern it established was worth more.
